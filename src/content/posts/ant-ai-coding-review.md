## 时间线：18 号

这一天应该是我正式开始准备的时间。因为我平时没有对编程 agent 写长篇完备 prompt 的习惯，所以我还是很担心在考试中写 prompt 的能力。另一方面，我也缺乏使用 agent 开发一个完整业务级项目的经验。[^note-1]

[^note-1]: 这里真正暴露的不是 prompt 长度问题，而是 agent 开发经验不足。前者是表层，后者更决定结果。

因此我觉得要从两个方面入手：

1. 针对考试本身的研究
2. 对 vibe coding 能力的研究

## 1. 对考试内容本身的研究

结合小红书内容，易得。人数众多，不可能完全人工评审。那么通过 AI 评审，我们可以得到一个显而易见的逻辑：提示词注入。[^note-2]

[^note-2]: “AI 评审”不等于“只靠提示词注入就能拿高分”。更稳的做法仍然是让结构、测试、文档彼此自洽。

此外，除了提示词注入，我们也很容易迎合 AI 的其他偏好，比如通过大量文本，吹嘘我们的代码效果很好，编写过多的说明文档，浮夸吹嘘代码效果、结构等。

## 2. 对 vibe coding 能力的研究

我意识到 prompt 是决定 agent 能力的主要内容。哪怕 agent 直出一个 prompt，也优于我每次只说一两句 prompt。

此外，我开始还需要解决 vibe coding 的通用原则，如测试先行、先 plan、基于第一性原理编码。

我让 Grok、GPT 查了一下有哪些必要的 skill、rules、`agents.md`。看了一大堆，结果 skill 与 rules 还是没用上，最后选择魔改 Everything Claude Code，并且对其提炼压缩，最后还汉化了一版。感觉不是完全用上了。[^note-3]

[^note-3]: 这类准备很容易陷入“收集工具幻觉”：看了很多规则，但没有变成现场可执行流程。

## 考试中

一开始还是很担心的。先把自己准备好的提示词复制了进来，然后发现了有 Git，直接克隆了自己提前准备的 agent 工具箱。便开始使用自己的工具箱给 AI 灌入知识，使用的是蚂蚁自家的 Code Fuse。这玩意我查了查，貌似 23、24 年就有，不过好像没什么热度。

agent 会默认读 `agents.md`，不过貌似没有深度读取整个工具库。一开始有些手忙脚乱，想开几个会话进行多线程，结果 AI 读同一个 `agents.md` 还有不同的反应，而且使用新会话时，旧会话会中止。最后调教了一个对文档理解不错的 agent，开始了 vibe coding。[^note-4]

[^note-4]: 平台行为本身就是考试的一部分。会话中断、上下文读取不稳定、默认读哪些文件，这些都属于实战约束。

途中按顺序遇到这些坑：

- agent 不新开 `app` 或 `src` 存储库运行
- 我的预设 prompt 中，没有对架构与技术选型的工作流。agent 直接按 `agents.md` 开干，缺乏全局规划，导致后面实现可选功能时没有预留接口，反复修改报错
- 过于要求最小 MVP，agent 直接实现了一个极其微小的产品作为启动[^note-5]
- 开新会话时会中断旧会话，无法多线程
- 甚至有 agent 串了上下文。在最后补全文档时，貌似直接把我还在准备另外一个会话的 prompt 给执行了
- 由于前期架构混乱导致的增量式开发，后面添加复杂功能，比如端到端加密模块时，就没有正确挂载上
- 测试直接使用了测试脚本，没有使用 `pytest`，没能判断正确性[^note-6]
- agent 没有把依赖项写好，甚至觉得自己本地原来有的就不用安装了

[^note-5]: 这里很关键。MVP 应该压缩实现复杂度，不该压缩水位到失去验收价值。

[^note-6]: 在这类题里，测试不只是辅助验证，而是交付物的一部分。没有稳定测试，很多“完成”其实只是表面完成。

## 时间线

### 0 - 5 分钟

安装工具库，准备会话上下文。

### 5 - 10 分钟

开摆了，找了一个能用的 agent 直接开始开发，导致了一个最简 MVP 实现。貌似就两个文件。

### 10 - 20 分钟

要求重新写 agent 中的内容，比如 `timeline.md`，让模型按严格表格执行，补充必需的功能实现。

### 30 - 45 分钟

继续补充任务要求的必需内容，并且让 agent 补充详细的测试。

### 45 - 60 分钟

意识到光看着 agent 干活没啥用，创建了一个人类 todo 文档，用于整理 agent 任务与构建 prompt。等待 agent 构建安全性要求内容。[^note-7]

[^note-7]: 这是一个正确动作。人类必须维护独立于 agent 的任务板，否则很容易被 agent 的局部产出牵着走。

### 60 - 75 分钟

AI 搓了一个简易的安全相关代码，我还没看明白。意识到可以添加 rules，先搞了一个第一性原理，以及要求其看代码的 rules。

### 75 - 100 分钟

让 agent 补充可选功能与智能算法部分，并且对安全算法进行补强。补充 rules，强调 TDD，要求 agent 先构思测试再写代码，并且要求 plan 先于编码。

### 100 - 110 分钟

构建两个 prompt：整理功能文档与安全性能说明。针对交付产物优化文本内容，并且避免在测试报告中暴露弱点。并且执行 prompt。[^note-8]

[^note-8]: 这一步说明你已经意识到：考试后半段比的不是继续加功能，而是整理交付物、控制叙事与暴露面。

### 110 - 120 分钟

执行上述 prompt，发现 `requirement` 文件写得一拖，要求优化。最后勉强赶上。
